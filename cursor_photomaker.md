ТЗ на проект PhotoMaker
1. Название проекта
PhotoMaker

2. Цель проекта
Разработать веб-сервис, который позволяет автоматически создавать маркетплейс-фотографии одежды на основе:

фотографии-референса модели,

изображения футболки с уже нанесённым принтом,

автоматически сгенерированного промпта для NanoBanana через ChatGPT.

Итог: пользователь получает готовые изображения, где внешний вид модели максимально сохраняет референс, но модель уже одета в выбранную футболку с нужным принтом.

3. Общая логика работы
Сервис должен работать по следующей схеме:

На сервере постоянно синхронизируются папки с Яндекс.Диска.

В одной папке находятся изображения футболок с уже нанесёнными принтами.

Референсные фотографии моделей могут:

уже храниться в системе,

подгружаться автоматически,

загружаться пользователем вручную с веб-страницы.

При первом появлении новой референсной фотографии:

она сохраняется в базе,

отправляется в ChatGPT,

ChatGPT генерирует текстовый промпт для NanoBanana,

промпт сохраняется и в дальнейшем повторно используется.

Пользователь на странице:

выбирает одну футболку с принтом,

выбирает один или несколько референсов моделей,

запускает генерацию.

Для каждого выбранного референса создаётся отдельная задача генерации.

В интерфейсе по каждому референсу отображаются:

превью,

статус,

время с момента отправки,

результат,

кнопка перегенерации,

кнопка скачивания.

Если задач несколько, должна быть кнопка «Скачать всё», которая скачивает все результаты одним архивом.

4. Основные сущности системы
4.1. Футболка с принтом
Изображение футболки, лежащее в отдельной папке, синхронизируемой с Яндекс.Диска.

Характеристики:

уникальное имя файла,

путь к файлу,

превью,

дата добавления,

статус доступности.

Пользователь может выбрать только одну футболку на одну сессию генерации.

4.2. Референс модели
Фотография человека, внешний вид которого нужно сохранить при генерации.

Характеристики:

оригинальный файл,

превью,

уникальный ID,

дата загрузки,

признак: загружен вручную / взят из базы,

автоматически сгенерированный ChatGPT-промпт,

история использования.

Если референс уже использовался ранее, пользователь может выбрать его повторно без повторной генерации промпта.

4.3. Промпт
Текст, который генерирует ChatGPT по референсной фотографии и который потом используется для NanoBanana.

Характеристики:

привязка к конкретному референсу,

дата генерации,

текст промпта,

версия промпта,

статус генерации.

Промпт должен сохраняться в базе и переиспользоваться при повторной работе с этим референсом.

4.4. Задача генерации
Отдельная задача на создание итогового изображения.

Характеристики:

ID задачи,

выбранная футболка,

выбранный референс,

используемый промпт,

статус,

время создания,

время отправки,

время завершения,

ссылка на результат,

возможность повторной генерации.

5. Пользовательский сценарий
5.1. Базовый сценарий
Пользователь открывает страницу PhotoMaker.

Вверху страницы он выбирает футболку с принтом.

Поиск по имени файла помогает быстро найти нужную футболку.

После выбора футболки пользователь добавляет один или несколько референсов моделей:

выбирает из уже существующих,

или загружает новые,

или загружает сразу несколько новых фото.

Для каждого выбранного референса на странице появляется отдельный блок.

Пользователь нажимает кнопку «Сгенерировать».

Для каждого блока запускается генерация изображения.

Пользователь видит статус по каждому блоку в реальном времени.

После завершения генерации в блоке появляется итоговая картинка.

Пользователь может:

скачать отдельный результат,

перегенерировать конкретный результат,

скачать все результаты архивом.

6. Функциональные требования
6.1. Синхронизация с Яндекс.Диском
Система должна автоматически синхронизировать сервер с Яндекс.Диском.

Требования:

синхронизация должна происходить без ручного участия пользователя;

при добавлении новых файлов на Яндекс.Диск они должны автоматически появляться на сервере;

новые файлы должны индексироваться системой;

если добавлен новый референс, он должен быть обработан автоматически;

если добавлена новая футболка с принтом, она должна стать доступной в поиске на сайте;

должна быть защита от дублирования файлов;

должна вестись проверка целостности синхронизации.

Желательно предусмотреть:

периодическую синхронизацию по расписанию,

логирование синхронизации,

ручную кнопку «Обновить с Яндекс.Диска» в админке.

6.2. Работа с референсами
Система должна поддерживать 3 режима работы с референсами:

1) Выбор из уже сохранённых
Пользователь выбирает ранее загруженный референс из базы.

2) Загрузка одного нового фото
Пользователь загружает новый файл вручную.

3) Загрузка нескольких новых фото
Пользователь может загрузить сразу несколько референсов, например 3–4 фото за раз.

Для нового фото обязательно:

сохранить файл,

создать превью,

отправить его в ChatGPT,

получить промпт,

сохранить промпт в базе.

6.3. Генерация промпта через ChatGPT
При первом появлении нового референса система должна автоматически отправлять изображение в ChatGPT.

ChatGPT должен вернуть промпт, содержащий описание внешности и ключевых признаков модели, пригодное для NanoBanana.

Требования:

промпт должен быть привязан к конкретному референсу;

при повторном использовании того же референса заново в ChatGPT фото не отправляется, используется уже сохранённый промпт;

должна быть возможность в будущем вручную пересоздать промпт;

промпт должен храниться в базе как отдельное поле.

6.4. Формирование финального промпта для NanoBanana
Перед отправкой задачи в NanoBanana система должна собирать итоговый промпт из нескольких частей:

описания модели из ChatGPT,

указания, что на модель нужно надеть выбранную футболку,

указания сохранить ракурс, позу, лицо, тип внешности и визуальную идентичность референса,

указания использовать именно выбранную футболку с принтом как одежду для модели,

при необходимости — технических параметров генерации.

Итоговый промпт должен формироваться автоматически.

6.5. Выбор футболки с принтом
На странице должен быть отдельный блок выбора футболки.

Требования:

пользователь может выбрать только один файл;

доступен поиск по имени файла;

по мере ввода должны подсказываться совпадения;

после выбора должна отображаться превьюшка футболки;

источник файлов — синхронизируемая папка на сервере.

6.6. Блоки генерации
После выбора футболки пользователь может добавлять блоки с референсами.

Каждый блок должен содержать:

превью референса,

имя файла,

статус подготовки промпта,

статус генерации,

таймер времени с момента отправки,

область показа результата,

кнопку перегенерации,

кнопку скачивания,

кнопку удаления блока до запуска генерации.

На странице должна быть кнопка «+» для добавления нового блока референса.

Количество блоков не ограничено логически, но может быть ограничено настройками системы.

6.7. Генерация изображений
После запуска генерации система должна отправить отдельную задачу в NanoBanana по каждому блоку.

Требования:

каждая задача независима;

ошибки в одной задаче не должны ломать остальные;

пользователь должен видеть прогресс по каждой задаче отдельно;

результат должен сохраняться на сервере;

результат должен быть доступен для скачивания.

6.8. Перегенерация
У каждого результата должна быть кнопка «Перегенерировать».

При нажатии:

создаётся новая задача генерации,

используется тот же референс,

используется та же выбранная футболка,

используется тот же базовый промпт,

новый результат сохраняется как новая версия либо заменяет старый — это надо предусмотреть в архитектуре.

Рекомендуемая логика:

хранить историю генераций, а пользователю показывать последнюю как основную.

6.9. Скачивание
Система должна поддерживать два режима скачивания:

1) Скачать одну картинку
Кнопка в каждом блоке.

2) Скачать все результаты
Одна общая кнопка на странице.

При скачивании всех результатов:

все успешные файлы архивируются в ZIP,

архив скачивается пользователю.

промпт получен,

Также рядом желательно показывать:

сколько времени прошло с момента отправки,

время завершения,

короткий текст ошибки, если она есть.

8. Структура страницы
8.1. Верхняя часть страницы
название проекта / страницы,

блок выбора футболки,

строка поиска по имени файла,

превью выбранной футболки.

8.2. Центральная часть
Список блоков генерации.

Каждый блок:

статус генерации,

результат,

кнопки действий.

8.3. Кнопки управления
добавить референс,

загрузить новый референс,

сгенерировать всё,

скачать всё.

Система должна хранить:

9.1. Референсы
оригинал файла,

превью,

путь,

9.2. Промпты
текст промпта,

9.3. Футболки
имя файла,

путь,

ID задачи,

11. Интеграции
11.1. Яндекс.Диск
Для синхронизации исходных файлов.

11.2. ChatGPT
Для генерации промпта по фотографии-референсу.

11.3. NanoBanana
Для генерации финального изображения модели в выбранной футболке.

12. Админ / служебная логика
Нужны служебные механизмы:

логирование всех ошибок,

логирование синхронизации,

логирование запросов в ChatGPT,

логирование запросов в NanoBanana,

повторный запуск зависших задач,

контроль очереди генераций,

защита от двойной отправки одной и той же задачи.

13. Нефункциональные требования
13.1. Производительность
страница должна быстро открываться;

поиск по футболкам должен работать без заметной задержки;

несколько генераций должны запускаться параллельно.

13.2. Надёжность
если одна задача упала, другие продолжают работать;

результаты не должны теряться после перезагрузки сервера;

статусы должны восстанавливаться после рестарта.

13.3. Масштабируемость
Нужно закладывать архитектуру с возможностью:

увеличивать число референсов на странице,

добавлять очередь задач,

добавлять новые типы одежды в будущем.

14. Что должен уметь пользователь
Пользователь должен иметь возможность:

найти и выбрать одну футболку с принтом;

выбрать уже существующий референс;

загрузить новый референс;

загрузить сразу несколько новых референсов;

добавить несколько блоков референсов на одну страницу;

запустить генерацию сразу по всем блокам;

смотреть статусы по каждому блоку;

перегенерировать отдельный результат;

скачать одну картинку;

скачать все картинки архивом.

15. Ожидаемый итог работы системы
На выходе пользователь получает удобный веб-интерфейс, где он:

выбирает нужную футболку с принтом,

подбирает несколько моделей-референсов,

запускает генерацию,

получает серию готовых фото для маркетплейсов,

может быстро пересобирать результаты и скачивать их по одному или все сразу.

7. Логика работы OpenAI
7.1. Когда вызывается OpenAI
Только когда появился новый референс, которого ещё нет в базе.

Сценарий:
Пользователь загружает фото.

Сервис считает hash.

Если такого hash нет:

файл сохраняется,

создаётся превью,

фото уходит в OpenAI,

OpenAI возвращает промпт,

промпт сохраняется в reference_prompts.

Если hash уже есть:

используем старую запись,

старый промпт не создаём заново.

7.2. Что хранить от OpenAI
Минимум:

сам текст промпта,

дату генерации,

статус,

ошибку, если была,

версию шаблона запроса.

8. Логика работы KAI / NanoBanana
8.1. Что уходит в KAI
На каждую генерацию должен формироваться финальный пакет:

выбранный референс,

выбранная футболка,

промпт от ChatGPT по модели,

общий шаблон промпта для NanoBanana.

8.2. Что делает generation_service
Получает shirt_id.

Получает reference_id.

Достаёт prompt из базы.

Собирает итоговый промпт.

Создаёт запись в generation_jobs.

Отправляет задачу в KAI.

Сохраняет внешний provider_job_id.

Периодически опрашивает статус.

Когда результат готов:

скачивает файл,

кладёт в data/results/original/,

делает preview,

создаёт запись в generation_results.

9. Синхронизация с Яндекс.Диском
9.1. Что синхронизируем
Папку с футболками.

Например:

Yandex Disk -> /photomaker/shirts/
Server -> /root/photomaker/data/shirts/original/
9.2. Как делать
Есть 2 нормальных варианта:

Вариант 1 — по API Яндекс.Диска
Сервис сам проверяет удалённую папку и докачивает новые файлы.

Вариант 2 — через mounted sync/cli
Если на сервере уже настроен Яндекс.Диск клиент, можно просто синкать папку локально.

Для архитектуры я бы закладывал через API, потому что:

понятнее контролируется,

можно логировать,

проще ловить новые файлы,

не зависишь от внешнего mounted-клиента.

9.3. Частота синхронизации
Через systemd timer, например:

каждые 2 минуты,
или

каждые 5 минут.

9.4. Что делает синхронизация
Получает список файлов с Яндекс.Диска.

Сравнивает с локальной БД и/или hash.

Скачивает новые файлы.

Создаёт preview.

Обновляет таблицу shirts.

Делает лог в sync.log.

10. Очередь задач
На старте можно не поднимать Redis/Celery, а сделать проще.

Стартовый вариант
Фоновая логика через:

SQLite,

systemd,

polling из Flask,

отдельные worker-функции.

Как это может работать
Вариант попроще:
после запуска генерации backend создаёт job;

отдельный поток или отдельный process обрабатывает задачу;

статусы пишутся в БД;

фронт опрашивает /api/generations/status.

Более надёжный вариант:
отдельный worker-скрипт, который крутится как service:

/root/photomaker/scripts/generation_worker.py
Он:

ищет jobs со статусом pending,

отправляет их в KAI,

проверяет processing,

сохраняет completed или failed.

Я бы сразу закладывал именно отдельный worker, это будет стабильнее.

16. Именование и UUID
Лучше не хранить всё только по исходным именам файлов.

Рекомендуемая схема:

в БД хранить оригинальное имя,

на диске хранить безопасное имя или UUID.

Например:

saved file name: 8f0f0d8f-28f1-4f5d-a1aa-112233445566.jpg
Это сильно упростит жизнь.

## 3. База данных (SQLite + SQLAlchemy)

### 3.1. Таблица `references`

- **id** (PK, int)
- **uuid** (str) — имя файла на диске (безопасное/UUID)
- **original_filename** (str) — исходное имя файла
- **original_path** (str) — путь в `data/references/original/`
- **preview_path** (str) — путь в `data/references/preview/`
- **file_hash** (str, unique) — hash для защиты от дублей
- **mime_type** (str)
- **width** (int)
- **height** (int)
- **created_at** (datetime)
- **updated_at** (datetime)
- **is_active** (bool)

### 3.2. Таблица `reference_prompts`

- **id** (PK)
- **reference_id** (FK → references.id)
- **prompt_text** (text)
- **prompt_version** (int)
- **status** (str) — `prompt_pending | prompt_processing | prompt_ready | prompt_failed`
- **error_message** (text, nullable)
- **generated_at** (datetime)
- **template_version** (str) — версия prompt-шаблона (опционально)
- **is_current** (bool) — текущая актуальная версия промпта

### 3.3. Таблица `shirts`

- **id** (PK)
- **filename** (str) — имя файла (как на Я.Диске)
- **original_path** (str) — путь в `data/shirts/original/`
- **preview_path** (str) — путь в `data/shirts/preview/`
- **file_hash** (str, nullable)
- **source_path** (str) — путь/ключ на Я.Диске
- **sync_status** (str) — `new | synced | removed | error`
- **created_at** (datetime) — когда впервые увидели файл
- **updated_at** (datetime)
- **is_active** (bool)

### 3.4. Таблица `generation_jobs`

- **id** (PK)
- **shirt_id** (FK → shirts.id)
- **reference_id** (FK → references.id)
- **reference_prompt_id** (FK → reference_prompts.id)
- **final_prompt** (text) — собранный промпт для Nano Banana
- **status** (str) — `pending | queued | submitted | processing | completed | failed | retrying | cancelled`
- **provider** (str) — `kie`
- **provider_job_id** (str) — task_id из Kie.ai
- **callback_received_at** (datetime, nullable)
- **created_at** (datetime)
- **started_at** (datetime, nullable)
- **finished_at** (datetime, nullable)
- **error_message** (text, nullable)
- **retry_count** (int)

### 3.5. Таблица `generation_results`

- **id** (PK)
- **job_id** (FK → generation_jobs.id)
- **result_path** (str) — `data/results/original/...`
- **preview_path** (str) — `data/results/preview/...`
- **file_size** (int)
- **width** (int)
- **height** (int)
- **created_at** (datetime)
- **is_current** (bool) — последняя актуальная версия для этого job

### 3.6. (Опционально) Таблица `generation_attempts`

Если нужна полная история перегенераций отдельно от `generation_results`.

## 4. Основные сервисы backend

- **`reference_service.py`**:
  - загрузка одного/нескольких референсов;
  - сохранение файла в `data/references/original/` (UUID-имя);
  - генерация превью в `preview/`;
  - расчёт `file_hash` и проверка дублей;
  - создание/получение записи `references`;
  - постановка задачи на генерацию промпта (через `prompt_service`/воркер).

- **`prompt_service.py`**:
  - проверка, есть ли уже актуальный промпт для референса;
  - вызов `openai_client` для новых референсов;
  - сохранение `reference_prompts`;
  - возможность пересоздать промпт вручную (новая версия).

- **`shirt_service.py`**:
  - индексация PNG из `data/shirts/original/` после синка;
  - генерация превью;
  - поиск по имени файла (`LIKE %q%`);
  - выдача данных для поиска/выбора футболки.

- **`generation_service.py`**:
  - сборка финального промпта:
    - текст модели (из `reference_prompts`);
    - указание надеть конкретную футболку;
    - сохранить ракурс/лицо/тип внешности;
    - тех. параметры Nano Banana;
  - создание записей `generation_jobs`;
  - вызов `kie_client` (создание задач);
  - обновление статусов по polling/ callback;
  - сохранение полученного результата в `data/results/` + превью;
  - работа с перегенерацией (новая job для того же reference+shirt).

- **`sync_service.py`**:
  - опрос Яндекс.Диска (через `yandex_disk_client`);
  - определение новых/удалённых/изменённых файлов;
  - скачивание PNG в `data/shirts/original/`;
  - создание превью;
  - обновление таблицы `shirts`;
  - логирование в `sync.log`;
  - хранение состояния в `data/sync_state/*.json`.

- **`status_service.py`**:
  - агрегация статусов по job;
  - форматирование времени с момента отправки;
  - подготовка кратких статусов/ошибок для UI.

## 5. Интеграции

### 5.1. OpenAI (`openai_client.py`)

- API-ключ и модель берутся из `.env`:
  - `OPENAI_API_KEY`
  - `OPENAI_MODEL=gpt-5.1`
- Основные операции:
  - приём path к референсу;
  - отправка изображения в OpenAI;
  - получение текстового описания/промпта;
  - логирование в `openai.log`.

### 5.2. Kie.ai / Nano Banana (`kie_client.py`)

- Конфиг из `.env`:
  - `KIE_API_KEY`
  - `KIE_BASE_URL=https://api.kie.ai`
  - `KIE_MODEL=nano-banana` (или nano-banana-pro / v2)
  - `KIE_CALLBACK_URL=https://your-domain.com/api/kie/callback`
- Основные операции:
  - отправка задачи генерации/редактирования:
    - `reference image` — референс модели;
    - `shirt image` — PNG футболки;
    - `prompt` — собранный финальный промпт;
  - получение `task_id` (**provider_job_id**);
  - polling статуса (fallback);
  - скачивание результата по готовому `task_id`.

### 5.3. Callback от Kie.ai

- **Endpoint**: `POST /api/kie/callback`
- Логика:
  - Kie.ai присылает `task_id`, `status`, ссылку на результат (или ошибку);
  - backend находит `generation_jobs` по `provider_job_id`;
  - обновляет статус и `callback_received_at`;
  - если `status=completed`:
    - скачивает файл;
    - кладёт в `data/results/original/`;
    - создаёт превью;
    - добавляет запись в `generation_results` (с `is_current=True`);
  - пишет лог в `kie.log`.

### 5.4. Yandex Disk (`yandex_disk_client.py`)

- Конфиг из `.env`:
  - `YANDEX_DISK_TOKEN`
  - `YANDEX_DISK_REMOTE_PATH=/photomaker/shirts`
- Функции:
  - получить список файлов в удалённой папке;
  - скачать новый файл;
  - (опционально) проверка ETag/модификации;
  - проброс ошибок в `sync.log`.

## 6. API-роуты

### 6.1. Страницы

- **GET `/`** — основная страница PhotoMaker (выбор футболки, референсов, запуск генерации).

### 6.2. Футболки (`/api/shirts`)

- **GET `/api/shirts/search?q=...`**
  - поиск по имени файла (по мере ввода).
- **GET `/api/shirts/<id>`**
  - данные по конкретной футболке.
- **POST `/api/shirts/reindex`** (внутренний/adm)
  - ручной перескан папки `data/shirts`.

### 6.3. Референсы (`/api/references`)

- **GET `/api/references`**
  - список уже сохранённых референсов (с пагинацией/фильтрами).
- **POST `/api/references/upload`**
  - загрузка одного нового фото.
- **POST `/api/references/upload-multiple`**
  - загрузка сразу нескольких фото.
- **GET `/api/references/<id>`**
  - данные по конкретному референсу.
- **POST `/api/references/<id>/regenerate-prompt`**
  - форсированное пересоздание промпта (новая версия).

### 6.4. Генерации (`/api/generations`)

- **POST `/api/generations/create`**
  - создание одной задачи генерации (shirt_id + reference_id).
- **POST `/api/generations/create-batch`**
  - создание пачки задач по одному `shirt_id` и списку `reference_id`.
- **GET `/api/generations/<job_id>/status`**
  - статус по конкретной задаче.
- **POST `/api/generations/<job_id>/regenerate`**
  - перегенерация (создаётся новая job для того же набора).
- **GET `/api/generations/list?shirt_id=...`**
  - (опционально) список задач по конкретной футболке.

### 6.5. Скачивание (`/download`)

- **GET `/download/result/<result_id>`**
  - скачать одну картинку.
- **GET `/download/job/<job_id>`**
  - скачать текущий результат конкретной задачи.
- **GET `/download/all?session=...`**
  - скачать ZIP всех результатов в рамках текущей сессии пользователя.

### 6.6. Callback от Kie.ai

- **POST `/api/kie/callback`**
  - описано в блоке 5.3.

## 7. Логика основной страницы (frontend)

- **Верх**:
  - название проекта/страницы;
  - блок выбора футболки:
    - строка поиска (AJAX → `/api/shirts/search`);
    - список подсказок;
    - превью выбранной футболки.
- **Центр**:
  - список блоков генерации:
    - превью референса;
    - имя файла;
    - статус промпта;
    - статус генерации;
    - таймер с момента отправки;
    - превью результата (если есть);
    - кнопки: «Перегенерировать», «Скачать», «Удалить блок до запуска».
- **Низ / управление**:
  - кнопка «+ Добавить референс»:
    - выбрать из базы;
    - загрузить новый;
    - загрузить несколько.
  - кнопка «Сгенерировать всё»:
    - отправляет shirt_id + список reference_id → `/api/generations/create-batch`.
  - кнопка «Скачать всё»:
    - запрос на ZIP-архив (создаётся через `zip_utils`).

- **Обновление статусов**:
  - периодический polling (JS, `setInterval`) по `/api/generations/<job_id>/status`;
  - обновление карточек без перезагрузки страницы.

### Кратко: реальные варианты для сервера

Для PhotoMaker есть по сути **3 рабочих способа** подключиться к Яндекс.Диску:

---

### 1. OAuth‑токен к HTTP API (то, что мы делаем сейчас)

- **Как работает**: один постоянный токен с правами на Диск → REST API → мы сами решаем, что и как синхронизировать.
- **Плюсы**:
  - полный контроль (логи, индексы, структура путей);
  - легко разворачивать на любом сервере;
  - не нужен GUI/клиент.
- **Минусы**:
  - нужно один раз корректно выдать токен с правами на Диск.
- **Подходит**: да, это идеальный серверный вариант для PhotoMaker.

---

### 2. WebDAV + логин/пароль / пароль приложения

- Яндекс.Диск поддерживает **WebDAV** (`https://webdav.yandex.ru`).
- Можно:
  - либо использовать **обычный логин/пароль Яндекс** (не очень безопасно),
  - либо создать **пароль приложения** и использовать его как «пароль» для WebDAV.
- На сервере тогда:
  - монтируешь WebDAV в файловую систему (через `davfs2`, `rclone` и т.п.),
  - или используешь библиотеки WebDAV-клиента.
- **Плюсы**:
  - видится как обычная файловая система; можно просто `rsync` / `cp`.
- **Минусы**:
  - сложнее надёжно настроить;
  - может быть медленнее/глючнее, чем прямой REST API;
  - всё равно нужно хранить чувствительные данные (логин/пароль или пароль приложения).

---

### 3. Официальный десктоп‑клиент Я.Диска, смонтированный в систему

- На сервер ставится официальный клиент Яндекс.Диска, который:
  - логинится в аккаунт,
  - синхронизирует локальную папку с облаком **автоматом**.
- Дальше PhotoMaker просто читает **обычную локальную папку**, а не ходит в API.
- **Плюсы**:
  - та самая «магическая» синхронизация, о которой ты думал.
- **Минусы**:
  - на прод‑серверы такое обычно не ставят (GUI-зависимости, обновления, контроль);
  - сложнее автоматизировать и мониторить.
